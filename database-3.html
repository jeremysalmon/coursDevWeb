Structure de base de données. Quelques règles

Quelques règles simples vous permettrons de mettre en place des structures robustes qui vous permettront de faire des applications solides.

  • Chaque table doit avoir un "identifiant". Cette identifiant doit être géré automatiquement par la base de données (par exemple avec un numéro qui s'incrémente à chaque insertion
  • Cet "identifiant" est utilisé dans la mise en place des relations (cardinalités) entre les tables. Par exemple dans le cas de la facture et des clients, comme chaque facture appartient à un seul client, nous pouvons stocker l'identifiant du client dans la table facture.
  • Vous devez nommez vos champs le plus clairement possible. Vous devez réussir à faire la balance entre longueur et lisibilité. Dans une table client "mail" sera aussi lisible que "adresseEmail". De la même façon "lastUpdate" sera plus lisible que "dupdt".
  • Si vous le pouvez, essayez de nommer vos champs en Anglais plutôt qu'en Français. Le nom de vos champs sera plus court.
  • Utilisez le Lower Camel Case pour nommer vos champs (premier mot en minuscule, mots suivants avec une majuscule en première lettre) (ex : lastUpdate, amount, clientCode). Vous devez respecter une logique afin de ne pas vous poser de questions sur "Comment j'ai appelé ce champs"
